Architecture

kcore is a clustered KVM virtualization platform built with Go, gRPC, and NixOS. This page describes the system topology, component responsibilities, and data flows.

Controller-First Architecture

All external clients вҖ” kctl and the Terraform provider вҖ” communicate exclusively with the controller on port 9090. Only the controller talks to node-agents on port 9091. This is a hard rule: kctl and Terraform never create direct connections to node-agents.

This single-entry-point design simplifies security (one TLS endpoint to expose), avoids split-brain coordination issues, and keeps node-agents invisible to end users.

Desired State vs Actual State

When a user submits a YAML manifest or Terraform resource, the controller stores it as desired state in its SQLite database. The controller never modifies desired state autonomously вҖ” it is the user's source of truth.

  • Desired state вҖ” the full resource spec including target lifecycle state (running, stopped, or deleted). Only changed by explicit user action.
  • Actual state вҖ” reported by node-agents: VM power state, actual node placement, resource usage. Updated via heartbeats and sync RPCs.

A reconciliation loop continuously compares desired to actual for every VM and takes action to converge вҖ” creating, starting, stopping, or deleting VMs as needed.

Component Overview

Component Port Role
Controller 9090 Cluster coordinator. Stores desired/actual state in SQLite, serves gRPC API, drives reconciliation.
Node-agent 9091 Per-node daemon. Manages local VMs via libvirt, reports state to controller, handles storage and networking.
kctl вҖ” CLI client. Sends gRPC requests to the controller. Supports declarative manifests (kctl apply -f) and imperative queries (kctl get vms).
Terraform вҖ” Terraform provider. Manages VMs as HCL resources via the same controller gRPC API.

Automator API

The Automator gRPC service (replacing the older NodeAdmin service) covers the full bare-metal node lifecycle. It is exposed on the same node-agent gRPC port (9091).

  • Hardware discovery вҖ” GetDisks, GetNics: enumerate local block devices and network interfaces.
  • OS installation вҖ” InstallNode: multi-disk, role-based NixOS installation with automatic partitioning.
  • Network configuration вҖ” ConfigureNetwork: bridges, bonds (LACP, active-backup), VLANs, DNS.
  • NixOS config push вҖ” UpdateNixConfig: deploy a new NixOS configuration to a node.
  • System update вҖ” UpdateSystem: trigger a nixos-rebuild switch on the node.

Bootstrap Flow

When a bare-metal node first boots from the kcore ISO, no TLS infrastructure exists yet. The bootstrap sequence establishes trust and configures the node:

  1. Node boots from the kcore ISO (minimal NixOS with node-agent in insecure mode).
  2. kctl init cluster generates the cluster CA and controller certificates on the operator's machine.
  3. kctl get disks and kctl get nics discover the node's hardware via the Automator API.
  4. kctl install node triggers NixOS installation on selected disks and issues TLS certificates for the node.
  5. After reboot, the node-agent starts with TLS enabled and registers with the controller.

CA Management

The cluster CA lives on the operator's machine at ~/.kcore/<cluster>/. The CA private key never leaves the operator's machine вҖ” it is not copied to nodes or the controller.

Node and controller certificates are issued during the installation process. Each component receives its own certificate signed by the cluster CA, enabling mutual TLS authentication across the cluster.

Multi-Disk Install

The InstallNode RPC supports multiple disks, each assigned a role:

  • os вҖ” root filesystem. Exactly one disk must have this role. Partitioned with boot, swap, and root partitions.
  • storage вҖ” LVM thin-pool for VM disks. Zero or more disks. Initialized as LVM physical volumes and added to a volume group for thin-provisioned VM storage.

Network Configuration

The ConfigureNetwork RPC supports complex networking topologies rendered into NixOS networking modules:

  • Bridges вҖ” for VM connectivity, attaching VMs to physical networks.
  • Bonds вҖ” LACP (802.3ad) and active-backup modes for link aggregation and redundancy.
  • VLANs вҖ” 802.1Q tagged interfaces for network segmentation.
  • DNS вҖ” nameserver configuration.

Configuration can be applied immediately using the applyNow flag, which triggers a NixOS rebuild after writing the network config.

gRPC Services

Controller services (port 9090):

  • RegisterNode, Heartbeat вҖ” node registration and health monitoring
  • SyncVmState вҖ” node-agents report VM state changes
  • VM lifecycle вҖ” CreateVm, StartVm, StopVm, DeleteVm, ListVms, GetVm
  • Node inventory вҖ” ListNodes, GetNode

Node-agent services (port 9091):

  • NodeCompute вҖ” VM operations (create, start, stop, delete domains via libvirt)
  • NodeStorage вҖ” disk image management, LVM thin-pool operations
  • NodeInfo вҖ” hardware and system information
  • Automator вҖ” bare-metal lifecycle (install, network, update)

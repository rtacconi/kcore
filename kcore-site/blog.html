Website Relaunch: Honest Positioning

We've relaunched the kcore website with clearer, more honest positioning. The previous version leaned too heavily on AI marketing language and implied a maturity level we haven't reached. That's been corrected.

kcore is now positioned as what it is: a modern Proxmox alternative in alpha. The messaging is straightforward — API-first, declarative, built for automation, not production ready. AI agents are first-class citizens in the platform design, but kcore is not an "AI platform."

New pages include this blog, a detailed roadmap, a technical architecture overview, and a FAQ that addresses maturity, Proxmox migration, and licensing honestly.

20 February 2026

Multi-Node Architecture Working

The controller + node-agent architecture is functional. Nodes self-register with the controller on boot and report their capabilities (CPU, memory, storage). The controller tracks cluster state in SQLite and routes VM operations to the appropriate node agent via gRPC.

Current topology: one controller (runs on your workstation or a management node) and N compute nodes. Each compute node runs NixOS with libvirtd and the kcore node-agent service.

What's next

  • mTLS between controller and agents (currently using insecure gRPC for development)
  • Automatic certificate provisioning on node registration
  • Node health checks and heartbeat monitoring
15 January 2026

Why We Started Building kcore

We've been running Proxmox in production for years. It works. But every time we needed to automate something — provision a VM from CI, template a cluster, reproduce an environment — we hit the same friction. The API is an afterthought to the web UI. The configuration model is imperative. The host OS is mutable Debian with all the drift that implies.

We wanted a virtualization platform that:

  • Has an API as its primary interface, not a web UI
  • Uses declarative configuration — describe what you want, not the steps to get there
  • Runs on an immutable host OS with atomic updates and instant rollback
  • Is built for automation from day one — scripts, Terraform, AI agents, whatever tools you use

Nothing like this existed, so we started building it. kcore is the result. It's very early — alpha, not production ready — but the foundation is solid: Go control plane, NixOS hosts, gRPC APIs, and a clean codebase with no legacy baggage.

If you're an operator who thinks about infrastructure this way, we'd like to hear from you.
